PRD — HADR Monitoring Agent

2026-07-08 · Decisions detailed in CONTEXT.md and docs/adr/0001–0012

1. What is the goal?

Let one user (the repo owner) know about humanitarian-relevant disasters without watching three websites. A single Python service polls GDACS, USGS, and ReliefWeb, deduplicates them into canonical events, and publishes them to a local web page the owner visits — triggered by modeled humanitarian impact (GDACS Orange/Red, PAGER yellow+, provisional M≥6.0 near populated land), not raw physical severity. Silence must be trustworthy: a feed outage shows as a degraded banner on the page, so no-alerts means genuinely calm.

2. What are the three vertical slices?

  1. USGS end-to-end — poll the summary feed → raw archive → store → trigger (provisional M≥6.0 + PAGER) → alert visible on the page. Ships the full dedup-ready schema from day one, proving the whole pipeline with one source.
  2. GDACS + real trigger policy — GDACS RSS ingestion, the full impact-based trigger (Orange/Red episode levels, hazard-scope filtering), and cross-source dedup working for real.
  3. ReliefWeb enrichment — behind a feature flag, enabled when the pre-approved appname arrives; adds editorial confirmation and GLIDE cross-references. Never triggers alerts.

3. What is the data model?

SQLite, three tables, plus a raw payload archive on disk (every fetched payload written verbatim before parsing — audit trail and replay-test corpus).

TableOne row perKey contents
events Canonical real-world disaster Hazard type, country/geometry, current alert level and trigger state, GLIDE (when known), timestamps
source_records Per-source claim, keyed by (source, source ID) Link to canonical event, latest parsed fields, update history, ID aliases (USGS re-keying), deletion flag
notifications Update recorded (shown on the web page) Event, transition type (new / escalation / confirmation / retraction), sent-at — feeds the page's updates list and drives the 30-min per-event coalescing

Dedup maps source records to canonical events: GLIDE exact match first, fuzzy fallback (hazard + country + ±48 h + geometry), merging conservatively — a false merge is worse than a missed one.

4. What's out of scope?

  • Multiple users, teams, ops centers — single owner, localhost only, no auth.
  • Push delivery (Telegram/email/Slack) and public hosting — the page is served locally and visited on demand.
  • Alerts for volcanoes and droughts (stored silently); wildfires below GDACS Red.
  • ReliefWeb as a trigger source — enrichment/confirmation only.
  • Rich content: maps, severity-history digests, per-source breakdowns on the page.
  • Global rate-cap / digest mode — per-event coalescing only, revisit with real storm data.
  • Serverless/cloud-cron deployment; anything beyond one systemd service on one always-on host.
  • Data pruning or retention policies — keep everything indefinitely.
