System View — HADR Monitor

How the pieces fit: three feeds are polled, archived, parsed, merged into canonical events, evaluated for humanitarian impact, and surfaced on a local web page. Data flows top to bottom. See CONTEXT.md and docs/adr/ for the why.

Canonical data model

  • events — one row per real-world disaster; current alert level, provisional/retracted flags, GLIDE, representative time/geometry.
  • source_records — each feed's claim about an event (+ source_aliases for USGS re-keying). Never overwritten across sources.
  • notifications — every recorded transition (new/escalation/confirmation/retraction); the web updates feed + coalescing basis.
  • feed_state — per-feed last success, failures, conditional-request validators; drives the staleness banner.

Invariants that hold the design together

  • Store raw before parsing. The archive is the source of truth for replay and audit.
  • Events are mutable claims. Triggers read canonical events, never raw feed items; each source's claim is preserved.
  • Impact over severity. GDACS episode level and PAGER drive alerts; magnitude only feeds the provisional fast path.
  • Silence is trustworthy. A dead feed surfaces as a banner — it is never indistinguishable from calm.
  • ReliefWeb never triggers. It only enriches events other feeds surfaced (GLIDE join).

Deployment: one always-on host. hadr run (poller) and hadr web (page) as systemd services; a timer regenerates dashboard.html at 08:30 Asia/Singapore. See deploy/.
